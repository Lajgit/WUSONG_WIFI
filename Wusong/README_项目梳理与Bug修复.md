# 武松球盘 Wusong_New_1.7 项目梳理与小游戏坑滞留钢珠修复

## 1. 项目概况

- MCU：STM32F407VGT6，主频配置为 168 MHz。
- 软件结构：裸机超级循环，无 RTOS。
- 主循环：`CommTask → FlashTask → Key_Task → Light_Task → CtrlTask → Mesg_Task`，各阶段穿插喂独立看门狗。
- Unity/安卓板通信口：USART1，115200、8N1。
- NFC/开锁板通信口：USART3，115200、8N1。
- 主协议帧：7 字节，`AA Code Data1 Data2 CRC_H CRC_L 55`。
- CRC：CRC16/MODBUS，初值 `0xFFFF`、多项式 `0xA001`；本工程在线路上按高字节、低字节顺序发送。

## 2. 目录职责

| 目录 | 职责 |
|---|---|
| `Core/` | STM32CubeMX 生成的时钟、GPIO、UART、TIM、IWDG、SPI 和中断入口 |
| `Drivers/` | STM32 HAL 与 CMSIS |
| `Lib/` | CRC、环形缓冲区、Flash、WS2812、SM16306S 等通用组件 |
| `Port/` | 对通信、设备、事件、按键、灯效的抽象封装 |
| `Task/` | 项目业务层：串口命令、机构控制、输入检测、灯光、消息上报 |
| `MDK-ARM/` | Keil/EIDE 工程配置和原有构建文件 |

## 3. 关键业务模块

### 3.1 `MainTask`

- `Main_Init()`：读取设置、初始化通信、机构、按键和灯光。
- `Main_Task()`：按固定顺序轮询全部任务。
- `Scene`：由 Unity 通过 `0x05` 命令控制球盘场景。

场景枚举：

| 值 | 场景 |
|---:|---|
| 0 | 设置 |
| 1 | 空闲 |
| 2 | 随机 |
| 3 | 投珠 |
| 4 | 公比/公共倍率 |
| 5 | 加注/Additional |
| 6 | 正常游戏 |
| 7 | 胜利 |
| 8 | 失败 |
| 9 | 开心游戏 1 |
| 10 | 开心游戏 2 |
| 11 | 结束 |

### 3.2 `CommTask`

负责 USART1/USART3 拆包、CRC 校验和命令分发。

Unity → 球盘主要命令：

| Code | Data | 功能 |
|---:|---|---|
| `0x01` | Data1:Data2 数量 | 吐珠 |
| `0x02` | Data1:Data2 数量 | 出卡 |
| `0x03` | - | 挡珠电磁阀 |
| `0x04` | Data1=1/5，Data2=亮度 | 面板灯/灯带亮度 |
| `0x05` | Data2=场景 | 设置场景 |
| `0x06` | 两字节位图 | 设置中奖通道和灯 |
| `0x07` | Data2=1/0 | 开心 30 秒输赢状态 |
| `0x08` | Data2=位图 | 按键灯位置 |
| `0x09` | Data1=1 | 清珠 |
| `0x09` | Data1=3 | 吐出剩余珠子和卡片；本次修复在此补充清理小游戏坑 |
| `0x0A` | - | 保存设置 |
| `0x0B` | - | 恢复默认设置 |
| `0x0C` | - | 手动触发左右小游戏气阀 |
| `0x0E` | - | 球盘复位 |
| `0x0F` | - | 开锁 |
| `0xFF` | - | 停止所有输出 |

### 3.3 `KeyTask`

- 12 路中奖微动检测。
- 5 路玩家按键检测。
- 3 路小键盘检测。
- 左右小游戏坑传感器检测：PE2、PE3，高电平代表坑内有球。
- 左右气阀输出：PC2、PC3。

钢珠进入小游戏坑并持续 200 ms 后，`Hole_LongCallback()`：

1. 向 Unity 上报 `0x14` 入坑消息；
2. 在正常游戏、加注、开心小游戏、设置场景中暂不弹球；
3. 在其他场景中自动触发左右气阀；
4. 设置 `Event_LeftHoleInside` 或 `Event_RightHoleInside`，直到钢珠离坑后才清除。

### 3.4 `CtrlTask`

维护各机构状态机：

- 吐珠电机：正转、超时、反转和重试；
- 卡片机：按反馈脉冲递减待出卡数量；
- 挡珠阀、锁阀、左右小游戏气阀；
- 气阀基础超时时间为 800 ms。

### 3.5 `InterruptTask`

处理中断输入：

- 投珠、投币；
- 吐珠反馈；
- 卡片反馈；
- UART 单字节中断接收并写入环形缓冲区。

### 3.6 `LightTask`

根据 `Scene` 控制：

- WS2812 面板灯；
- 6 路呼吸灯；
- SM16306S 中奖通道灯；
- 部分场景切换时停止挡珠阀或左右小游戏阀。

## 4. 故障根因

故障不是气动阀硬件异常，而是“持续占位传感器 + 只处理首次入坑”的状态遗漏。

原流程：

1. 钢珠进入拔河或打老虎坑，PE2/PE3 变高；
2. `Hole_LongCallback()` 首次执行并设置 `Event_LeftHoleInside`/`Event_RightHoleInside`；
3. 此时处于 `SCENE_PLAYING`、`SCENE_ADDITIONAL` 等场景，代码按设计不启动气阀；
4. 缺卡后玩家加卡，钢珠始终没有离坑，传感器一直保持高电平；
5. 因坑内事件位仍为已置位状态，后续轮询不会重新执行首次入坑动作；
6. Unity 发送“吐出剩余”命令 `0x09 / Data1=0x03` 时，原代码只恢复吐珠电机和卡片机，没有检查小游戏坑；
7. 因此钢珠继续滞留，需要人工拿出。

关键点：场景变化或加卡不会制造新的传感器上升沿，必须在“吐出剩余”时主动读取坑位当前电平。

## 5. 修复内容

### 修改文件

1. `Task/Inc/CtrlTask.h`
2. `Task/Src/CtrlTask.c`
3. `Task/Src/KeyTask.c`
4. `Task/Src/CommTask.c`

### 新增统一函数

```c
void MiniGameBall_EjectIfOccupied(void);
```

函数行为：

- 直接读取 PE2、PE3 当前电平；
- 左右任一坑确实有球时才处理；
- 仅当两路气阀均为 `IDLE` 且触发计数均为 0 时启动；
- 按项目原有方式同时触发左右气阀；
- 不清除坑位事件位，钢珠离坑后仍由原 `Hole_ReleaseCallback()` 清除，保持原状态机一致。

### 修复触发点

`CommTask.c` 的 `0x09 / Data1=0x03` 分支新增：

```c
MiniGameBall_EjectIfOccupied();
```

原正常入坑后的非游戏场景自动弹球逻辑也改为调用同一个函数，避免左右分支判断条件不一致。

## 7. 建议测试流程

### 用例 A：复现并验证主故障

1. 烧录重新编译的修复固件。
2. 进入正常游戏，使钢珠落入拔河或打老虎坑。
3. 确认 Unity 收到左坑或右坑 `0x14` 上报。
4. 保持钢珠在坑内，制造无卡状态。
5. 添加卡片并执行返回游戏/吐出剩余流程。
6. 确认 Unity 发出 `AA 09 03 00 0E F1 55`。
7. 预期 PC2、PC3 产生气阀驱动，钢珠被弹出。
8. 钢珠离坑后 PE2/PE3 应恢复低电平，坑位事件位由原释放回调清除。

### 用例 B：左右坑分别测试

分别让钢珠停在左坑和右坑，重复用例 A，两个坑都必须能够在“吐出剩余”时弹出。

### 用例 C：无球时不误动作

1. 确保 PE2、PE3 均为低电平。
2. 发送“吐出剩余”。
3. 预期左右气阀不动作，吐珠和出卡原流程不受影响。

### 用例 D：正常游戏行为回归

1. 正常游戏中钢珠进入小游戏坑。
2. 在未执行“吐出剩余”前，仍应保持原设计：球不会立即自动弹出。
3. 小游戏正常结束时，原有 Unity `0x0C` 或场景流程仍可正常控制气阀。

### 用例 E：阀门忙状态保护

气阀正在动作时再次发送“吐出剩余”，预期不会重置当前阀门计时或重复启动。

## 8. 代码审查中发现的其他风险（本修复包未改）

### 8.1 `0x07` 缺少 `break`

`CommTask.c` 中 `case 0x07` 处理开心游戏输赢后会继续进入 `case 0x08`，从而把 `ButtonLight_Position` 同时改成 `Data2`。如果不是有意复用，应补 `break`。

### 8.2 `0x0D` 会直接落入 `0x0E` 复位

`case 0x0D`“重新绑卡”没有代码也没有 `break`，收到该命令会继续执行 `System_Reset()`。需要结合 Unity 是否使用该命令确认需求。

### 8.3 左右气阀触发计数命名和实际脉冲数需台架确认

当前 `Valve_TriggerCount=1` 的状态机包含一次超时后的再次启动行为，看起来可能产生两段气阀脉冲。这可能是为增强弹球力度而设计，不建议在未做示波器/实际机构确认前修改。

### 8.4 修复包不包含新编译固件

原压缩包内的 HEX/BIN 是修改前生成的旧产物。修复包已移除这些生成目录，必须在 Keil/EIDE 中重新 Rebuild 后再烧录。已对修改的 C 文件做 Cortex-M4 目标语法检查，但当前环境没有原项目使用的 ARMCC 5，未生成可刷写固件。
