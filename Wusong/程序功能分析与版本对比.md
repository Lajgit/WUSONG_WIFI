## 1. 工程总体结构

本工程是基于 STM32F407VGTx 的 HAL/CubeMX + Keil MDK 项目。程序采用主循环轮询任务架构，`Core` 完成芯片外设初始化，`Task` 承载业务逻辑，`Port` 封装硬件抽象，`Lib` 提供通用组件和外设驱动。

| 目录/文件 | 作用 | 关键位置 |
| --- | --- | --- |
| `Core/Src/main.c` | HAL 初始化、系统时钟、外设初始化、进入主循环 | `main()`：第 69 行；外设初始化：第 93-107 行 |
| `Core/Src/gpio.c` | GPIO 引脚方向、上下拉、初始电平配置 | `MX_GPIO_Init()`：第 42 行 |
| `Core/Src/usart.c` | UART4、USART1、USART2、USART3 初始化与 MSP 配置 | `MX_UART4_Init()`：第 33 行；`MX_USART1_UART_Init()`：第 62 行；`MX_USART2_UART_Init()`：第 91 行；`MX_USART3_UART_Init()`：第 120 行 |
| `Core/Src/tim.c` | PWM、基础定时器、灯光/电机相关定时器配置 | `MX_TIM1_Init()`：第 36 行；`MX_TIM2_Init()`：第 110 行；`MX_TIM3_Init()`：第 168 行；`MX_TIM5_Init()`：第 222 行；`MX_TIM7_Init()`：第 280 行；`MX_TIM9_Init()`：第 313 行 |
| `Core/Src/spi.c` | SPI1、SPI2 初始化 | `MX_SPI1_Init()`：第 31 行；`MX_SPI2_Init()`：第 63 行 |
| `Core/Src/dma.c` | DMA 控制器初始化 | `MX_DMA_Init()`：第 39 行 |
| `Core/Src/iwdg.c` | 独立看门狗初始化 | `MX_IWDG_Init()`：第 30 行 |
| `Drivers` | STM32 HAL/CMSIS 驱动库 | ST 官方库文件，业务逻辑不在此处 |
| `MDK-ARM` | Keil 工程文件、启动文件、RTE 配置 | `Wusong_New_1.7.uvprojx`、`Wusong_New_1.7.uvoptx` |

## 2. 主流程与任务调度

程序入口在 `Core/Src/main.c`：

- 第 69 行：`main()` 入口。
- 第 86 行：调用 `SystemClock_Config()` 配置系统时钟。
- 第 93-107 行：初始化 GPIO、DMA、SPI、TIM、USART、IWDG 等外设。
- 后续进入主循环，周期调用业务任务。

业务任务入口在 `Task/Src/MainTask.c`：

| 功能 | 文件位置 | 说明 |
| --- | --- | --- |
| 系统复位 | `Task/Src/MainTask.c:16` | `System_Reset()` 关闭中断并触发 NVIC 软件复位 |
| 系统指示灯 | `Task/Src/MainTask.c:22` | `SystemLight_Task()` 每 500 ms 翻转 `User_LED` |
| 模块初始化 | `Task/Src/MainTask.c:32` | `Main_Init()` 依次初始化 Flash、通信、设备、按键、灯光 |
| 主任务循环 | `Task/Src/MainTask.c:41` | `Main_Task()` 轮询通信、Flash、按键、灯光、控制、消息任务，并多次喂狗 |

`Main_Task()` 调度顺序为：

1. `CommTask()` 通信收包处理。
2. `FlashTask()` 参数保存。
3. `Key_Task()` 输入检测。
4. `Light_Task()` 灯光场景刷新。
5. `CtrlTask()` 电机、卡机、电磁阀等执行器状态机。
6. `Mesg_Task()` 待发送事件消息处理。
7. `SystemLight_Task()` 系统灯闪烁。

## 11. 业务代码修改细节

### 11.1 新增小游戏洞口占球统一弹出函数

新增位置：

- 声明：`Task/Inc/CtrlTask.h:42`
- 实现：`Task/Src/CtrlTask.c:287`

函数行为：

1. 读取 `Hole_Left_Pin`、`Hole_Right_Pin` 两个洞口传感器。
2. 如果左右洞口都没有球，直接返回。
3. 如果左右阀触发次数都是 0，且左右阀状态都为 `DEVICE_STATE_IDLE`，同时调用：
   - `Valve_Start(&Left_Valve, Valve_TriggerCount)`
   - `Valve_Start(&Right_Valve, Valve_TriggerCount)`

这个修改把“洞口有球时触发左右电磁阀弹出”的判断集中到一个函数中，避免不同入口各自写不完整的判断。

### 11.2 通信命令 `0x09/0x03` 增加占球弹出

修改位置：`Task/Src/CommTask.c:121-125`

fixed 版在“吐出剩余”处理里，除了停止吐珠和出卡，还调用：

```c
MiniGameBall_EjectIfOccupied();
```

影响：

- 旧版只执行 `Hoolle_Output(&Motor_Hoolle1, 0)` 和 `Card_Output(&Card, 0)`。
- fixed 版会额外检查小游戏左右洞口，如果有球残留且左右阀空闲，则触发左右阀弹出。

### 11.3 洞口长按回调改为统一弹出逻辑

修改位置：

- 左洞口：`Task/Src/KeyTask.c:145-156`
- 右洞口：`Task/Src/KeyTask.c:159-168`

旧版在左洞口和右洞口回调中分别写了类似逻辑：

- 判断左右阀 `TriggerCount` 是否为 0。
- 只检查当前侧阀门是否 idle。
- 直接启动左右阀。

fixed 版改为调用 `MiniGameBall_EjectIfOccupied()`：

```c
MiniGameBall_EjectIfOccupied();
```

影响：

- 左右洞口使用同一套弹出判断。
- 只有实际检测到左右任一洞口仍有球时才触发左右阀。
- 同时要求左右阀都处于 `DEVICE_STATE_IDLE`，比旧版只检查单侧阀门更保守，降低正在动作时重复触发的风险。

### 11.4 修改目的归纳

fixed 版的核心修复点是：小游戏洞口存在残留球时，提供更可靠的自动弹出逻辑。

具体改动效果：

1. “吐出剩余”命令现在会处理小游戏洞口残留球。
2. 左右洞口长按检测不再各自维护弹出逻辑，改为统一函数。
3. 弹出动作增加左右洞口占用检测，避免无球时误触发。
4. 弹出动作增加左右阀同时空闲判断，避免左右阀动作中被重复启动。
